### Detailed Plan

#### Architecture Decision: esbuild Bundle for Production, Source for Dev

**Production (published npm package):**
- Bundle `apps/server/` source tree into a single CJS file `cli/app/server.js` using esbuild
- Externalize only native modules (`better-sqlite3`, `sql.js`) — resolved at runtime via `NODE_PATH` (existing `buildEnvWithRuntime()`)
- All `@9router/*` packages and npm deps (`hono`, `bcryptjs`, etc.) are bundled inline
- Output is CJS so Node.js runs it without needing `"type": "module"` in `cli/app/`

**Dev (monorepo checkout):**
- Set `9ROUTER_DEV=1` when running from the monorepo checkout
- Server path resolves to `path.join(__dirname, '..', 'apps', 'server', 'src', 'index.js')`
- `cwd` set to `path.join(__dirname, '..', 'apps', 'server')` so relative `./app.js`, `./config.js`, `./routes/*` resolve correctly
- Dependencies resolved via npm workspaces (monorepo root `node_modules` has `@9router/*` symlinked)
- Extra NODE_PATH entry for monorepo root `node_modules` passed via `extraNMPaths` to `buildEnvWithRuntime()`

**Why CJS output for the bundle:**
- The spawned child process runs `node server.js`. Without a `package.json` with `"type": "module"` in `cli/app/`, Node treats `.js` as CJS
- esbuild handles ESM → CJS conversion cleanly for the server's dependency tree (static imports become `require()`, dynamic `import()` become `Promise.resolve().then(() => require(...))`)
- This avoids adding a `package.json` to `cli/app/`
- `better-sqlite3` and `sql.js` are dynamic-imported inside try/catch — esbuild correctly preserves these as runtime `require()` lookups

**Why externalize native modules only:**
- `better-sqlite3` ships a compiled `.node` binary — esbuild cannot inline native code
- `sql.js` ships a `.wasm` file loaded dynamically — esbuild cannot bundle wasm
- Both are already managed by the existing runtime system: postinstall installs them into `~/.9router/runtime/node_modules`, and `buildEnvWithRuntime()` adds that path to `NODE_PATH`
- Bundling all other deps keeps the published package self-contained — no runtime npm install needed for server deps

### Risks and Mitigations

| Risk | Severity | Likelihood | Mitigation |
|------|----------|------------|------------|
| **esbuild ESM→CJS conversion fails** for some dependency (e.g., `@hono/node-server` uses ESM-only APIs) | High | Low | Test the bundle early. If ESM→CJS fails, switch to ESM output format + add `"type": "module"` to `cli/app/package.json` or use `.mjs` extension |
| **Dynamic imports in SQLite adapters break** after bundling (esbuild converts `await import(...)` to `require()` which changes error-throwing behavior) | Medium | Low | The try/catch blocks in `driver.js` will still catch errors. Test with and without better-sqlite3 installed |
| **Monorepo `node_modules` not found** in dev mode (wrong relative path calculation) | Medium | Medium | Path is `path.join(__dirname, '..')` from `cl/cli.js` → `9router/`. Then `path.join(root, 'node_modules')`. Symlinked packages are under this path. Test on macOS, Linux, Windows |

| **Published package size** — bundled server is ~5-10MB, much smaller than old ~200MB | Low | Low | Size reduction is a feature, not a risk. MITM bundle still adds ~2MB |
| **require('./package.json') in CLI breaks** if cwd changes | Low | Low | The `require("./package.json")` at `cli/cli.js:45` resolves relative to `__dirname` (the file), not `cwd`. This already works correctly |
